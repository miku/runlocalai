# Use Cases for Local AI Models at Universities

Practical use cases for running local LLMs/VLMs — organized by audience. Many
overlap across roles; the grouping highlights where the most immediate value
lies.

---

## Everyone (students, staff, faculty)

### Writing and language

* **Grammar and style checks** — paste a paragraph, get suggestions; works well
  with a system prompt like "act as a strict copy editor for academic German/English"
* **Translation and bilingual drafting** — draft in one language, translate to
  another; useful for international students and multilingual publications
* **Email drafting and reformulation** — turn bullet points into a polite,
  professional email; adjust tone (formal/informal)
* **Summarization** — condense long documents, meeting notes, or policy texts
  into key points

### Daily productivity

* **Brainstorming and ideation** — generate lists of ideas, counter-arguments,
  or alternative framings for any topic
* **Explaining concepts** — "explain X like I'm a first-year student" or "give
  me the graduate-level version"
* **Calendar/task text generation** — draft agendas, to-do lists, structured
  outlines from rough notes

### Privacy-sensitive tasks

* **Working with confidential data** — local models never leave the machine;
  suitable for unpublished manuscripts, grant proposals, personnel matters,
  patient data (with appropriate safeguards), exam questions
* **GDPR compliance** — no data sent to third-party APIs; relevant for
  processing student information, survey responses, or HR documents

---

## Students (undergraduate and graduate)

### Studying and learning

* **Flashcard and quiz generation** — feed lecture notes or a textbook chapter,
  generate Q&A pairs for self-testing
* **Concept mapping** — ask the model to outline relationships between concepts
  in a course module
* **Paper comprehension** — paste an abstract or methods section, ask for a
  plain-language explanation
* **Exam preparation** — generate practice questions on a topic, then check
  your own answers against model output

### Academic writing

* **Thesis outlining** — generate a structured outline from a research question
  and a handful of bullet points
* **Paraphrasing and rewording** — rephrase a paragraph to avoid repetition or
  improve clarity (not plagiarism — improving your own drafts)
* **Citation formatting assistance** — convert between citation styles, or
  generate a rough bibliography entry from a DOI or title
* **Feedback simulation** — "critique this paragraph as if you were a reviewer"

### Coding and technical work

* **Code explanation** — paste a function, get a line-by-line walkthrough
* **Debugging help** — describe the error, get candidate fixes
* **Boilerplate generation** — scaffolding for scripts, data pipelines, LaTeX
  templates
* **Data wrangling** — generate pandas/R snippets for cleaning, reshaping, or
  plotting data

## PhD students and researchers

### Literature and research

* **Literature screening** — feed abstracts (e.g. from a Zotero export), ask
  the model to classify relevance, extract methods, or identify gaps
* **Related work drafts** — given a set of paper summaries, draft a related
  work section or a comparison table
* **Research question refinement** — iterate on a research question by asking
  for alternative framings, scope narrowing, or operationalization suggestions
* **Methodology brainstorming** — "what are common approaches to study X in
  field Y?"

### Data and analysis

* **Qualitative coding assistance** — feed interview transcripts or open-ended
  survey responses, ask for initial thematic codes (human validates)
* **Statistical method selection** — describe your data and research question,
  get suggestions for appropriate tests or models
* **Synthetic data generation** — create realistic but fake datasets for
  testing analysis pipelines before using real data
* **Log and experiment notebook parsing** — summarize lab notebooks or
  experimental logs

### Writing and publication

* **Cover letter drafting** — for journal submissions, grant applications, job
  applications
* **Reviewer response drafting** — paste reviewer comments, generate structured
  response drafts point-by-point
* **Abstract generation** — from a full paper draft, generate a condensed abstract
* **Title brainstorming** — generate candidate titles from an abstract

### Code and reproducibility

* **Script documentation** — generate docstrings, README sections, or usage
  examples for research code
* **Pipeline prototyping** — quickly scaffold data processing or ML training
  pipelines
* **Notebook cleanup** — turn a messy Jupyter notebook into a cleaner, annotated
  version

## Faculty and instructors

### Teaching

* **Exercise and exam generation** — generate problem sets, multiple-choice
  questions, or case studies for a given topic and difficulty level
* **Rubric drafting** — create grading rubrics from learning objectives
* **Syllabus assistance** — outline a course schedule given topics and weeks
* **Personalized feedback templates** — generate individualized feedback
  scaffolds for student submissions (human reviews and edits)

### Research supervision

* **Proposal feedback** — "read this proposal abstract and list potential
  weaknesses"
* **Literature gap identification** — given a set of abstracts, identify what
  is missing or underexplored
* **Presentation coaching** — paste slide text, get suggestions for clarity and
  flow

### Administrative (faculty-side)

* **Committee document drafting** — reports, evaluations, recommendation letters
* **Meeting minutes from rough notes** — turn bullet points into structured minutes
* **Policy summarization** — condense lengthy university regulations into actionable points

## Administrative staff

### Communication

* **Template generation** — form letters, notification emails, FAQ entries for
  students or applicants
* **Multilingual communication** — draft announcements in German and English
  simultaneously
* **Tone adjustment** — make a technical policy text more accessible for a
  student audience

### Document processing

* **Form and report summarization** — condense accreditation documents, audit
  reports, or policy updates
* **Data extraction from text** — pull structured information (dates, names,
  figures) from unstructured documents
* **Procedure documentation** — turn informal process knowledge into step-by-step
  guides

### Operations

* **FAQ generation** — given a policy document, generate a list of likely
  questions and answers
* **Event planning support** — generate checklists, schedules, and draft
  invitations
* **Job posting drafting** — generate or refine job descriptions from a list of
  requirements

## Cross-cutting: local model advantages

| Concern | Local model benefit |
|---|---|
| Data privacy | Nothing leaves your machine |
| Cost | No per-token billing after hardware investment |
| Availability | Works offline, no API outages |
| Customization | Fine-tune or swap models freely |
| Reproducibility | Pin a specific model version for consistent results |
| Experimentation | Try many models, system prompts, parameters without cost pressure |

## Agentic workflows: beyond chat

The real step up from "chatting with a model" is giving the model **tools** —
the ability to search, read files, execute code, call APIs. This is where local
AI starts to feel like an assistant rather than a text box.

### Open-source agentic tools (Claude Code-like, work with Ollama)

| Tool | What it does | Install | Best for |
|------|-------------|---------|----------|
| [**Aider**](https://github.com/Aider-AI/aider) | Git-aware terminal coding agent; auto-commits, architect mode | `pip install aider-chat` | Coding, easiest to demo |
| [**Goose**](https://github.com/block/goose) (Block) | Autonomous agent: installs, edits, executes, tests; native MCP support | `pipx install goose-ai` | General agentic tasks |
| [**OpenCode**](https://opencode.ai) | Go-based terminal coding agent with TUI, LSP, 75+ providers | single binary | Coding, lightweight |
| [**Fabric**](https://github.com/danielmiessler/fabric) | Modular CLI for curated AI prompts ("patterns") against any input | `pip install fabric-ai` | Non-coding tasks, text processing |
| [**Open Interpreter**](https://github.com/openinterpreter/open-interpreter) | Natural-language computer control; executes Python/shell/JS | `pip install open-interpreter` | General automation |
| [**Continue**](https://continue.dev) | VS Code / JetBrains extension, fully local via Ollama | IDE marketplace | IDE users |

Quick start example with Aider:

```
ollama pull qwen2.5-coder:32b
aider --model ollama/qwen2.5-coder:32b
```

### Agentic frameworks (for building custom agents)

| Framework | Style | Local model setup | Good for |
|-----------|-------|-------------------|----------|
| [**smolagents**](https://huggingface.co/docs/smolagents) (HuggingFace) | Code-first, minimal | `LiteLLMModel("ollama_chat/qwen2.5:32b")` | Fastest "hello world" agent |
| [**CrewAI**](https://docs.crewai.com) | Multi-agent roles ("researcher", "writer") | via LiteLLM proxy | Multi-agent demos |
| [**LangGraph**](https://langchain-ai.github.io/langgraph/) | Graph-based state machine | any OpenAI-compatible endpoint | Complex pipelines |

### MCP (Model Context Protocol) — connecting models to real tools

MCP lets a model call external tools (search APIs, databases, file systems)
through a standardized protocol. Ollama doesn't natively speak MCP, but bridges
exist:

* [**ollmcp**](https://github.com/jonigl/mcp-client-for-ollama) — TUI client
  connecting Ollama to any MCP server (agent mode, multi-server, streaming)
* [**ollama-mcp-bridge**](https://github.com/patruff/ollama-mcp-bridge) —
  Python proxy that auto-injects MCP tools into Ollama chat requests
* **Goose** — has native MCP + native Ollama support, acts as the bridge
  naturally

### MCP servers relevant to university audiences

These are the "tools" the agent can use — services people already interact
with, now accessible to a local model:

| MCP Server | Connects to | Use case |
|-----------|-------------|----------|
| [**paper-search-mcp**](https://github.com/openags/paper-search-mcp) | arXiv, PubMed, Semantic Scholar, Google Scholar | Literature search |
| [**Scientific-Papers-MCP**](https://github.com/benedict2310/Scientific-Papers-MCP) | arXiv, OpenAlex, PMC, CORE, bioRxiv | Paper discovery |
| [**zotero-mcp**](https://github.com/54yyyu/zotero-mcp) | Local Zotero library (semantic search, annotations, PDFs) | Reference management |
| [**PapersGPT for Zotero**](https://github.com/papersgpt/papersgpt-for-zotero) | Zotero plugin with MCP support | Library Q&A |
| filesystem MCP | Local files | Read/summarize/organize documents |
| web-search MCP | DuckDuckGo, Brave, etc. | General research |

### Which models actually work for tool calling?

Agentic workflows need models that reliably format tool calls. Not all do.

| Model | Size | VRAM (Q4) | Tool calling | Notes |
|-------|------|-----------|-------------|-------|
| **Qwen 2.5 Coder 32B** | 32B | ~22 GB | Excellent | Best local tool caller, handles multi-step chains |
| **Qwen 3 32B** | 32B | ~22 GB | Very good | Hybrid thinking mode, 256K context |
| Llama 3.1 8B | 8B | ~6 GB | Decent | Minimum viable for simple tool demos |
| Qwen 3 8B | 8B | ~6 GB | Good | Punches above its weight |
| Mistral/Ministral 3B | 3B | ~2 GB | Basic | Smallest with any tool-calling ability |

Important: Ollama defaults to 2-4K context — for agentic work, increase to at
least 16K:

```
OLLAMA_NUM_CTX=16384 ollama serve
```

### Prerequisites for agentic demos

**Software:**
1. Ollama (model runtime)
2. One CLI agent tool (Aider or Goose — pick one)
3. Python 3.10+ (for smolagents, MCP servers)
4. Optional: Open WebUI, Continue extension

**Hardware (minimum for agentic work):**
* 32B model (recommended for reliable tool calling) needs ~24 GB VRAM or 32 GB
  unified memory (Apple Silicon)
* An 8B fallback works on 16 GB RAM but will stumble on complex tool chains
* Pre-pull models before the workshop — downloading 20 GB on conference Wi-Fi
  is not fun

**Models to pre-pull:**
```
ollama pull qwen2.5-coder:32b    # primary: coding + tool calling
ollama pull qwen3:8b              # fallback: lighter, still decent
ollama pull llava:13b             # vision model for image demos
```

### Demo ideas that wow (just a little)

**1. "Find me papers and add them to Zotero" (3 min live)**
Connect Goose (Ollama backend) to paper-search-mcp + zotero-mcp. Ask:
"Find 5 recent papers on [audience-suggested topic] from OpenAlex, summarize
each in one sentence, and add them to my Zotero library." The model searches,
summarizes, and the papers appear in Zotero in real time.

**2. "Fix this bug" — live coding agent (5 min)**
Open a small repo with a deliberate bug. Run `aider --model
ollama/qwen2.5-coder:32b`. Describe the bug in natural language. Watch it find
the file, edit the code, and auto-commit the fix. Git log shows the diff.

**3. "Summarize this PDF in my style" — Fabric one-liner (2 min)**
```
cat paper.pdf | fabric --pattern extract_wisdom --model ollama/qwen3:8b
```
Instant structured extraction: key ideas, quotes, references, actionable items.
Swap `extract_wisdom` for `summarize`, `analyze_claims`, `write_essay`, etc.

**4. "Build me a research agent" — smolagents in 10 lines (5 min)**
Live-code a Python script that creates an agent with web search + file write
tools, asks it to research a topic, and writes a structured summary to a file.
The audience sees the agent's thinking process step by step.

**5. "Read my handwriting" — vision model (2 min)**
Take a photo of handwritten notes (or a whiteboard). Feed to `ollama run
llava:13b` with "transcribe this." The model reads handwriting — imperfect but
impressive. Follow up with "now clean up the grammar and organize as bullet
points."

**6. "What does this policy say about me?" — local document Q&A (3 min)**
Paste a university regulation (Prüfungsordnung, Studienordnung) into the
context. Ask concrete questions: "How many times can I retake an exam?" "What
is the deadline for thesis registration?" Answers come from the document, not
the model's training data. Privacy: the text never leaves the room.

## Approaches to try in the workshop

1. **Ollama + terminal** — simplest setup; `ollama run llama3` and start chatting
2. **Open WebUI** — browser-based chat interface on top of Ollama; good for
   non-technical users
3. **System prompts** — customize behavior: "You are a strict German grammar
   checker. Only output corrections."
4. **Document Q&A (RAG-lite)** — paste text into context, ask questions about it;
   no vector DB needed for short documents
5. **Vision models** — feed images (diagrams, handwritten notes, charts) to a
   VLM like LLaVA or Qwen-VL for description or OCR
6. **Code completion** — use a local model as a backend for VS Code (Continue) or
   other editors
7. **Batch processing with scripts** — use the Ollama API from Python/curl to
   process many files (e.g. classify 100 abstracts)
8. **Agentic coding** — Aider or Goose with Ollama for live bug fixing and
   code generation
9. **MCP tool use** — connect a local model to academic APIs (paper search,
   Zotero) via MCP servers
10. **Custom agents** — smolagents for building a simple research agent in
    10 lines of Python
